# Pattern Library

## Purpose
This document turns the approved translated inspirations into a small reusable design grammar for future districts, professions, projects, NPC interactions, creature encounters, and parent-child session structure. Use it to combine existing verbs, entities, information structures, scope layers, and visible consequence without inventing new ontology, widening Milestone 1, or turning references into a wishlist.

## Core Rule
A valid pattern is a repeatable combination of existing verbs, entities, scope layers, and visible consequence; if it needs new ontology or widens the current milestone, it is not ready.

## Pattern Types
- **District / profession grammar**: reusable structure for adding new local work without replacing the core loop.
- **Civic restoration grammar**: reusable structure for turning contribution into visible public change.
- **Social request-and-reflection grammar**: reusable structure for making villagers socially useful without turning them into chatter.
- **Creature-and-wonder grammar**: reusable structure for place-bound life, observation, and care.
- **Co-op session grammar**: reusable structure for parent-child play pacing and shared consequence.

## Pattern Entries
### Place -> Work -> Visible Consequence
- **What it is**: A district or profession pattern where a meaningful `Place` anchors repeated gathering, making, trade, maintenance, or contribution and the result changes life, workflow, access, or atmosphere in ways players can revisit.
- **What problem it solves**: It gives future districts and professions a stable additive shape so new content can feel local and distinct without inventing a separate game grammar for each expansion.
- **Where it belongs in the roadmap**: `Milestone 3 / Phase 3` as the main additive-expansion pattern after Orchard and Bridge is proven and later districts need to reuse the same permanent verbs.
- **What it must not become**: Profession sprawl, lore-only expansion, or a replacement tier that makes Orchard and Bridge feel fake, obsolete, or beginner-only.

### Project Site -> Project Stage -> Civic Asset
- **What it is**: A public-work pattern where shared contribution advances a `Project Site` through visible `Project Stage` changes until it becomes a durable `Civic Asset` or `Access Structure`.
- **What problem it solves**: It keeps civic design child-legible, place-based, and production-safe by making public contribution change something players can point to in the world.
- **Where it belongs in the roadmap**: `Milestone 1 / Phase 1` as the baseline civic pattern for Orchard and Bridge, then later reuse for future restorations and district projects.
- **What it must not become**: An invisible town score, a donation void, or cosmetic-only progress that never changes access, workflow, beauty, or shared life.

### Need -> Commission -> Feedback
- **What it is**: A social pattern where a villager or specialist expresses a concrete need, that need becomes a bounded `Commission`, and the outcome returns concise `Feedback` tied to what the player did.
- **What problem it solves**: It gives NPCs and specialists a repeatable systemic role so requests stay relational and useful instead of turning into quest logs, lore terminals, or idle chatter.
- **Where it belongs in the roadmap**: A simple feedback seed belongs in `Milestone 1 / Phase 1`, a fuller specialist chain belongs in `Milestone 3 / Phase 3`, and richer judgment around partial perspective belongs in `Milestone 4 / Phase 4`.
- **What it must not become**: Quest-log filler, affection grind, or chatter-first dialogue that hides weak systems behind more text.

### Observation -> Care -> Place Response
- **What it is**: A creature-and-wonder pattern where observation reveals a living presence or condition, care or patience changes the relationship, and the place responds in a visible way over time.
- **What problem it solves**: It keeps creature encounters tied to stewardship, habitat, timing, and quiet mystery so creatures deepen place meaning instead of pulling the game into a separate collection ladder.
- **Where it belongs in the roadmap**: A light form belongs in `Milestone 3 / Phase 3`, with deeper durable creature-place consequence in `Milestone 4 / Phase 4` and `Milestone 5 / Phase 5`.
- **What it must not become**: A collection ladder, a combat loop, or a rarity chase that eclipses craft, society, and restoration.

### Wonder -> Plan -> Shared Consequence
- **What it is**: A parent-child session pattern where an immediately readable action creates a short plan and then resolves into visible shared consequence that both players can understand in the same world.
- **What problem it solves**: It gives private co-op a repeatable play arc so sessions feel collaborative and legible without requiring shared-everything systems, admin-heavy controls, or abstract coordination mechanics.
- **Where it belongs in the roadmap**: Personal-plus-civic session framing belongs in `Milestone 1 / Phase 1`, and the first true household trade-off belongs in `Milestone 2 / Phase 2`.
- **What it must not become**: Shared-everything blur, forced co-dependence, or host-admin complexity that turns parent-child structure into permission management.

## Pattern Fit By Phase
| Milestone / Phase | Pattern fit |
| --- | --- |
| `Milestone 0 / Phase 0` | Translation and review vocabulary only. Use the library to filter proposals, not to authorize new features. |
| `Milestone 1 / Phase 1` | Only `Project Site -> Project Stage -> Civic Asset`, simple `Need -> Commission -> Feedback` reflection, and `Wonder -> Plan -> Shared Consequence` in personal-plus-civic form are active. |
| `Milestone 2 / Phase 2` | `Wonder -> Plan -> Shared Consequence` gains real household meaning through the first shared trade-off. |
| `Milestone 3 / Phase 3` | `Place -> Work -> Visible Consequence`, fuller `Need -> Commission -> Feedback`, and light `Observation -> Care -> Place Response` become valid. |
| `Milestone 4+ / Phases 4-5` | Bounded judgment, rumor, record, trust, and deeper durable creature-place consequence can layer onto the same patterns without replacing them. |

## Pattern Anti-Patterns
- Treating the library as a content wishlist instead of a small reusable grammar.
- Inventing new ontology, namespaces, or scope layers because a pattern "feels special."
- Letting consequence collapse into hidden meters instead of visible place change.
- Letting co-op read as either two solo loops nearby or a shared-everything blur.
- Letting NPCs become chatter, exposition, or affection grind instead of requests, feedback, and consequence.
- Letting creature charm, rarity, or collection pressure outrun craft, stewardship, and society.
- Letting later content make Orchard and Bridge feel like disposable tutorial debris.

## Recommendation
For each future proposal, choose one primary pattern and at most two supporting patterns from this library. Then name the roadmap phase fit, the visible consequence, the active scope layer, and what the proposal is explicitly not so the team can reject hidden scope growth early.

## Final Rule
If a proposal cannot be expressed as a combination of these patterns and the current ontology, defer it instead of inventing new grammar.
