# Production Docs

## Purpose
These docs control change during inception and early pre-production. Use them to understand current state, active risks, milestone timing, the Codex operating contract, and the rules that keep the repo stable as it grows.

## Start Here
- Read `current_state.md` first for the active milestone, current priority, top active risks, and approved next baby steps.
- Then read `codex_workflow.md` for the repo-specific Codex execution contract.
- If you are working on the current Milestone 2 household slice or its next design pass, read `milestone_1_closure.md`, `milestone_2_plan.md`, `milestone_2_first_shape_review.md`, and `milestone_2_household_slice_closure.md`, then pair them with `docs/design/household_pantry_contract.md` and `docs/design/household_reserve_contract.md`.
- If you are implementing Orchard and Bridge, then read `milestone_1_implementation_order.md` and `milestone_1_checklist.md` before feature work begins.

## What Each Doc Controls
- `current_state.md`: the active milestone, current priority, what is ready, what is not ready, and the next valid steps
- `codex_workflow.md`: the repo-specific Codex read order, output contract, and post-task sync rule
- `milestone_1_closure.md`: the frozen verified Milestone 1 baseline and the safe boundary before household expansion
- `milestone_2_plan.md`: the Milestone 2 household scope boundary and first-trade-off planning anchor
- `milestone_2_first_shape_review.md`: what the pantry-only first shape proved and why reserve-aware withdrawal became the next narrow step
- `milestone_2_household_slice_closure.md`: the current verified pantry-plus-reserve household baseline and the active unresolved household-purpose question
- `milestone_1_implementation_order.md`: the approved Orchard and Bridge build sequence
- `milestone_1_checklist.md`: the readiness, build, and review gate for Milestone 1 implementation
- `development_operating_model.md`: how ideas move from docs to implementation without drift
- `risk_register.md`: the active failure modes, early warning signs, and reinforcement priorities
- `roadmap.md`: milestone intent, banned scope, and proof of success
- `production_scale_phase_plan.md`: the ordered medium-granularity phase sequence between roadmap intent and backlog work
- `backlog.md`: near-term actionable work aligned to the roadmap and current risks
- `decision_log.md`: accepted repo-level decisions and their consequences
- `id_policy.md`: stable canonical ID rules for long-lived content
- `save_compatibility_policy.md`: save-version, content-version, migration, and deprecation discipline
- `content_schema_reference.md`: compact design-facing schema rules for early content records
- `title_strategy.md`: repo slug, codename, and working-title guidance

## Working Rule
Canon decides meaning. Design defines the current slice. Production controls change.
